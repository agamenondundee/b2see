1. Purpose

This Secure Development Policy (the “Policy”) defines how Cloudax Ltd. (“Cloudax”) embeds information security into the design, build, test, deployment, and maintenance of its conversational AI platform and supporting systems.

The Policy gives effect to the Annex A controls covering the secure development lifecycle and supports Cloudax’s commitments to customers and regulators.

ISO 27001:2022 references: A.5.1 Policies for information security; A.8.25 Secure development life cycle

2. Scope

This Policy applies to all software, systems, infrastructure-as-code, and AI components developed, modified, or operated by Cloudax, including:

  • Cloudax’s production conversational AI platform and customer-facing applications.
  • Internal services, tools, and integrations.
  • Infrastructure-as-code, configuration management, and CI/CD pipelines.
  • AI and ML components, including model training, fine-tuning, evaluation, and inference.
  • Open-source contributions made by Cloudax personnel in the course of their work.

It applies to all personnel, contractors, and partners who design, develop, test, deploy, or operate Cloudax software.

ISO 27001:2022 references: A.8.25 Secure development life cycle

3. Definitions

  • SDLC: software development lifecycle.
  • CI/CD: continuous integration and continuous delivery / deployment.
  • SBOM: software bill of materials — a list of components and dependencies that make up a software artefact.
  • SAST / DAST / SCA: static, dynamic, and software composition application security testing.

4. Principles

  • Security by design: security is considered from the earliest stages of design, not retrofitted.
  • Shift left: security feedback is provided as early in the lifecycle as possible.
  • Defence in depth: controls are layered across code, build, deployment, and runtime.
  • Least privilege: services run with the minimum permissions required.
  • Reproducible builds: builds and deployments are automated, deterministic, and traceable.
  • Continuous improvement: feedback from incidents, testing, and reviews drives ongoing improvement of practices and tooling.

ISO 27001:2022 references: A.8.25 Secure development life cycle; A.8.27 Secure system architecture and engineering principles

5. Roles and Responsibilities

  • Engineering Lead / Director of Technology: owns this Policy and the secure development programme.
  • Information Security Lead: provides assurance, threat modelling support, and security testing oversight.
  • Engineers: design, develop, and test in accordance with this Policy and supporting standards.
  • Reviewers: perform code review, ensure security checks are not bypassed, and enforce branch protection rules.

ISO 27001:2022 references: A.5.2 Roles and responsibilities; A.8.25

6. Secure Development Lifecycle

Cloudax operates a secure development lifecycle covering:

  • Requirements: security and privacy requirements identified for each material change.
  • Design: threat modelling for high-impact components and changes.
  • Implementation: secure coding standards, peer code review, and automated checks in pull requests.
  • Verification: SAST, SCA, secret scanning, and (for in-scope services) DAST and penetration testing.
  • Release: protected pipelines, controlled deployments, and signed artefacts.
  • Operate: vulnerability management, monitoring, and incident response.
  • Sunset: secure decommissioning and data deletion.

ISO 27001:2022 references: A.8.25 Secure development life cycle; A.8.32 Change management

7. Security Requirements and Threat Modelling

  • Security and privacy requirements are captured for material changes alongside functional requirements.
  • Threat modelling is performed for new high-impact services, significant architectural changes, and high-impact AI features. Outputs are documented and tracked to mitigation.
  • Customer contractual security requirements are identified and traced through to design and implementation.

ISO 27001:2022 references: A.8.26 Application security requirements; A.8.27 Secure system architecture and engineering principles

8. Secure Coding Standards

  • Cloudax maintains secure coding standards covering input validation, output encoding, authentication, authorisation, error handling, logging, and language-specific anti-patterns.
  • Standards reflect well-established industry guidance (e.g. OWASP ASVS, OWASP Top 10, CWE Top 25) adapted to Cloudax’s technology stack.
  • Compliance with secure coding standards is enforced through linting, code review, and automated checks.

ISO 27001:2022 references: A.8.28 Secure coding

9. Source Code Management

  • Source code is stored only in approved repositories with SSO and MFA required for access.
  • Branch protection rules require code review by a separate person, status checks, and signed commits where supported, before merge to protected branches.
  • Direct push to protected branches is prohibited.
  • Secrets must not be committed to source control. Secret scanning is enabled and detected secrets are rotated immediately.
  • Repository access is reviewed at least quarterly.

ISO 27001:2022 references: A.8.4 Access to source code; A.5.18 Access rights

10. Build, CI/CD, and Artefact Integrity

  • Builds are performed by automated CI/CD pipelines with controlled access.
  • Build environments are ephemeral and isolated.
  • Build artefacts are signed and stored in approved registries with access controls.
  • A software bill of materials (SBOM) is generated for production artefacts and retained.
  • Pipelines are configured to block deployments where mandatory security checks fail.

ISO 27001:2022 references: A.8.31 Separation of development, test and production environments; A.8.32 Change management; A.8.30 Outsourced development

11. Application Security Testing

  • Static application security testing (SAST) and software composition analysis (SCA) run on every change.
  • Dynamic application security testing (DAST) is performed for in-scope services on a defined cadence.
  • Independent penetration testing is performed on the production platform at least annually and after material change.
  • Findings are triaged, prioritised, and tracked to closure within timeframes appropriate to severity.

ISO 27001:2022 references: A.8.29 Security testing in development and acceptance; A.8.8 Management of technical vulnerabilities

12. Vulnerability Management and Dependency Management

  • Open-source and third-party dependencies are reviewed before introduction and tracked in the SBOM.
  • Vulnerability advisories are monitored. Vulnerable dependencies are remediated within timeframes appropriate to severity.
  • Production runtimes, container images, and infrastructure components are scanned and patched on defined cadences.

ISO 27001:2022 references: A.8.8 Management of technical vulnerabilities; A.8.9 Configuration management

13. Secrets Management

  • Secrets (credentials, keys, tokens, certificates) are stored only in approved secret management services.
  • Secrets are not stored in source code, configuration files, container images, environment variables in unsecured systems, or developer notes.
  • Secrets are rotated on a defined cadence and immediately on suspected compromise.
  • Access to production secrets is restricted, logged, and reviewed.

ISO 27001:2022 references: A.8.24 Use of cryptography; A.5.17 Authentication information

14. Environment Separation and Test Data

  • Development, test, staging, and production environments are logically and (where appropriate) physically separated.
  • Production data must not be used in non-production environments unless it has been masked, anonymised, or otherwise sanitised in line with the Information Classification and Handling Procedure (ISMS-PRC-004).
  • Access between environments follows the principle of least privilege.

ISO 27001:2022 references: A.8.31 Separation of development, test and production environments; A.8.33 Test information

15. Change Management and Release

  • Changes to production are managed in accordance with the Change Management Procedure (ISMS-PRC-009).
  • Each change is peer-reviewed, automatically tested, and authorised through the CI/CD pipeline.
  • Emergency changes follow a defined expedited process with retrospective review.
  • Releases are recorded with traceability from requirement to deployment.

ISO 27001:2022 references: A.8.32 Change management

16. Cryptography

  • Cryptographic protocols, algorithms, and key lengths follow current good-practice baselines and are documented in the Cryptography and Key Management Procedure (ISMS-PRC-005).
  • Personal data and Confidential or Restricted information is encrypted in transit and at rest.
  • Cryptographic keys are managed through approved key management services with rotation, access controls, and logging.

ISO 27001:2022 references: A.8.24 Use of cryptography

17. AI and Machine Learning Components

Cloudax’s products include AI and ML components. The following requirements apply in addition to the general SDLC controls:

  • Training, fine-tuning, and evaluation data are classified, governed, and where personal data is involved subject to appropriate Article 6 / Article 9 / Article 28 controls.
  • Model risks (e.g. prompt injection, data exfiltration via model outputs, hallucination, jailbreaking) are explicitly addressed in threat modelling and tested in security assessments.
  • Use of third-party model providers is governed by the Supplier Security Policy (ISMS-POL-004) and the AI Risk Management Procedure (ISMS-PRC-013).
  • AI components include logging that supports incident investigation and post-hoc analysis, while respecting data protection requirements.
  • Where AI outputs are used in customer-facing decisions, appropriate human oversight, evaluation, and disclosure are designed in.

ISO 27001:2022 references: A.8.25 Secure development life cycle; A.8.26 Application security requirements; A.5.34 Privacy and protection of PII; A.5.23 Cloud services

18. Outsourced Development

Where Cloudax outsources development, the supplier is engaged in accordance with the Supplier Security Policy (ISMS-POL-004) and is required to:

  • Apply security practices at least equivalent to those defined in this Policy.
  • Make their development environment, source code controls, testing, and incident management arrangements available to Cloudax for assurance.
  • Ensure that intellectual property, confidentiality, and data protection clauses in the supplier contract are honoured.

ISO 27001:2022 references: A.8.30 Outsourced development; A.5.19–A.5.22 Supplier relationships

19. Logging and Monitoring of Development Activity

  • Material development activities (commits to protected branches, deployments to production, secret access, administrative actions) are logged and retained in line with the Records Retention Schedule.
  • Logs are protected against tampering and reviewed in support of audit, incident response, and continual improvement.

ISO 27001:2022 references: A.8.15 Logging; A.8.16 Monitoring activities

20. Review and Maintenance

This Policy is reviewed at least annually and on material change. Material changes are approved by the Managing Director and recorded in Annex B.

ISO 27001:2022 references: Cl. 7.5, 9.3; A.5.1

21. Acknowledgement

All personnel involved in development are required to acknowledge this Policy on commencement, on material change, and on request.
