1. Purpose

This Policy defines how Cloudax Ltd. embeds information security into the design, build, test, deployment, and maintenance of its conversational AI platform and supporting systems.

ISO 27001:2022 references: Linked controls: A.8.25, A.8.26, A.8.27, A.8.28, A.8.29, A.8.30, A.8.31, A.8.33

2. Scope

This Policy applies to all software, systems, infrastructure-as-code, and AI components developed, modified, or operated by Cloudax.

3. Definitions

  • SDLC: software development lifecycle.
  • CI/CD: continuous integration and continuous delivery / deployment.
  • SBOM: software bill of materials.
  • SAST/DAST/SCA: static, dynamic, and software composition application security testing.

4. Principles

  • Security by design.
  • Shift left.
  • Defence in depth.
  • Least privilege for services.
  • Reproducible, traceable builds.
  • Continuous improvement informed by incidents and testing.

5. Roles and Responsibilities

  • Chief Technology Officer: owns this Policy and the SDLC programme.
  • Information Security Manager: provides assurance and threat modelling support.
  • Engineers: design, develop, and test in accordance with this Policy.
  • Reviewers: perform code review and enforce branch protection.

6. Secure Development Lifecycle

Cloudax’s SDLC covers:

  • Requirements: security and privacy requirements identified per material change.
  • Design: threat modelling for high-impact components.
  • Implementation: secure coding standards, peer review, automated checks.
  • Verification: SAST, SCA, secret scanning, DAST, penetration testing.
  • Release: protected pipelines, controlled deployments, signed artefacts.
  • Operate: vulnerability management, monitoring, incident response.
  • Sunset: secure decommissioning.

ISO 27001:2022 references: Linked control: A.8.25 Secure development life cycle

7. Security Requirements and Threat Modelling

  • Security and privacy requirements are captured for material changes.
  • Threat modelling is performed for new high-impact services and high-impact AI features.
  • Customer contractual requirements are traced to design and implementation.

ISO 27001:2022 references: Linked control: A.8.26 Application security requirements

8. Secure Coding Standards

  • Standards reflect OWASP ASVS, OWASP Top 10, CWE Top 25, adapted to Cloudax’s stack.
  • Compliance is enforced through linting, code review, and automated checks.

ISO 27001:2022 references: Linked control: A.8.28 Secure coding

9. Source Code Management

  • Source code repositories require SSO and MFA.
  • Branch protection requires peer review before merge to protected branches.
  • Direct push to protected branches is prohibited.
  • Secret scanning is enabled; detected secrets are rotated immediately.
  • Repository access is reviewed at least quarterly.

10. Build, CI/CD, and Artefact Integrity

  • Builds use automated CI/CD pipelines with controlled access.
  • Build environments are ephemeral and isolated.
  • Build artefacts are signed and stored in approved registries with access controls.
  • An SBOM is generated for production artefacts and retained.
  • Pipelines block deployments where mandatory security checks fail.

11. Application Security Testing

  • SAST and SCA run on every change.
  • DAST is performed on a defined cadence.
  • Independent penetration testing is performed at least annually and after material change.
  • Findings are triaged and tracked to closure within timeframes appropriate to severity (POL-014).

12. Security Testing in Development and Acceptance

  • Acceptance criteria for material changes include security tests.
  • Security tests cover authentication, authorisation, input validation, error handling, and known vulnerability classes.

ISO 27001:2022 references: Linked control: A.8.29 Security testing in development and acceptance

13. Secure Architecture and Engineering Principles

  • Designs follow established secure architecture principles — minimisation, isolation, fail-safe defaults, complete mediation, least privilege, defence in depth.
  • Architecture decisions are documented; high-impact decisions are reviewed by the Information Security Manager.

ISO 27001:2022 references: Linked control: A.8.27 Secure system architecture and engineering principles

14. Test Information

  • Production data is not used in non-production environments without masking, pseudonymisation, or anonymisation (POL-016) or unless customer contract permits.
  • Test data is selected, generated, and protected so that test conditions adequately exercise the system without exposing real personal data.

ISO 27001:2022 references: Linked control: A.8.33 Test information

15. Separation of Environments

  • Development, test, staging, and production environments are logically and where appropriate physically separated (POL-022).
  • Access between environments follows least privilege.

ISO 27001:2022 references: Linked control: A.8.31 Separation of development, test and production environments

16. AI and Machine Learning Components

Additional requirements apply to AI/ML components:

  • Training, fine-tuning, and evaluation data are classified and governed.
  • Model risks (prompt injection, data exfiltration via outputs, hallucination, jailbreaking) are addressed in threat modelling and tested.
  • Use of third-party model providers is governed by POL-006.
  • AI components include logging that supports investigation while respecting data protection.
  • Where AI outputs are used in decisions, appropriate human oversight, evaluation, and disclosure are designed in.

17. Outsourced Development

  • Where development is outsourced, suppliers apply security practices at least equivalent to this Policy and POL-006.

ISO 27001:2022 references: Linked control: A.8.30 Outsourced development

18. Logging of Development Activity

  • Material development activities (commits, deployments, secret access, administrative actions) are logged in line with POL-019 and retained per POL-009.

19. Review and Maintenance

This Policy is reviewed at least annually and on material change.

20. Acknowledgement

All personnel involved in development are required to acknowledge this Policy on commencement, on material change, and on request.
