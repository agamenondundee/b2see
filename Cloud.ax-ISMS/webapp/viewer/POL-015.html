1. Purpose

This Policy defines how Cloudax Ltd. manages the configuration of its hardware, software, services, and networks to ensure they are configured securely and consistently.

ISO 27001:2022 references: Linked control: A.8.9 Configuration management

2. Scope

This Policy applies to all configuration items in the Cloudax ISMS scope, including endpoints, cloud infrastructure, container platforms, applications, network devices, identity providers, and security tooling.

3. Definitions

  • Configuration: the settings of a system that affect its behaviour and security posture.
  • Baseline: a documented, approved set of configurations representing the secure default for a class of system.
  • Drift: an unintended divergence between the configured state of a system and its baseline.

4. Principles

  • Secure by default: systems are configured to minimise attack surface.
  • Codified: configuration is expressed as code where possible (Infrastructure-as-Code).
  • Version-controlled: configurations are stored in version control with peer review.
  • Auditable: configuration drift and changes are detected and reviewed.

5. Roles and Responsibilities

  • Chief Technology Officer: owns this Policy and the configuration management framework.
  • Information Security Manager: approves baselines; reviews drift and exceptions.
  • System owners: maintain configurations to baseline; remediate drift.

6. Configuration Baselines

  • Baselines are documented for each class of system (endpoints, cloud services, container images, network devices).
  • Baselines align with established benchmarks (e.g. CIS Benchmarks) where available, adapted for Cloudax.
  • Baselines are reviewed at least annually and on material change to threats or technology.

7. Infrastructure-as-Code

  • Cloud and platform configuration is defined as code in approved repositories.
  • Changes follow the same review and approval as application code (POL-024, POL-025).
  • Manual changes (“click-ops”) in production are avoided; where unavoidable, they are reconciled into code promptly.

8. Configuration Drift Detection

  • Cloud and infrastructure drift is detected through automated tooling.
  • Endpoint configuration is enforced through MDM.
  • Drift is investigated, remediated, and where appropriate logged as an event under PRO-027.

9. Standard Builds for Endpoints

  • Cloudax-issued endpoints are deployed from a standard build defined in MDM.
  • Standard build includes encryption, endpoint protection, browser hardening, and approved software.
  • Deviations require approval and are time-bound.

10. Cloud Service Configuration

  • Cloud service configuration follows POL-007 Cloud Services Policy and provider hardening guidance.
  • Configuration is monitored continuously and changes are tracked.

11. Network Device Configuration

  • Where network devices are operated by Cloudax, configurations are version-controlled and backed up.
  • Default credentials are changed; remote management is restricted.

12. Configuration Changes

  • Changes to configurations follow POL-025 Change Management Policy.
  • Emergency changes are documented retrospectively.

13. Backup of Configurations

  • Configurations are stored in version control and replicated as part of POL-018 Backup Policy.

14. Configuration Audit

  • Configuration is audited as part of internal audit (PROG-AUDIT-001).
  • Findings are addressed through corrective action.

15. Review and Maintenance

This Policy is reviewed at least annually and on material change.

16. Acknowledgement

All personnel involved in configuration are required to acknowledge this Policy on commencement, on material change, and on request.
